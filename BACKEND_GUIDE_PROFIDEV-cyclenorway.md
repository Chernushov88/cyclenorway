# BACKEND_GUIDE_PROFIDEV-cyclenorway.md

Цей документ визначає стандарти роботи з Gutenberg-блоками та ACF для проекту Cyclenorway.

## 1. Реєстрація блоків
- **Автоматизація**: Система автоматично реєструє блоки шляхом сканування директорії `blocks/*/block.json` через `RecursiveIteratorIterator`.
- **Правило**: Не реєструвати блоки вручну у `functions.php`. Створення нової папки `blocks/NewBlock/` з файлом `block.json` достатньо для появи блоку в адмінці.

## 2. Структура блоку (Обов'язкові файли)
Кожен блок повинен містити:
- `block.json`: Метадані блоку (apiVersion: 3).
- `template.php`: Фронтенд-рендеринг.
- `styles.scss`: Головний файл стилів блоку.
- `style-editor.scss`: Стилі для редактора Gutenberg.
- `styles.scss`: (опційно) Місце для запису CSS.
- `script.asset.php`: Депенденції JS.

## 3. Робота з ACF (Поля)
- **JSON-синхронізація**: Всі поля мають бути експортовані в `config/acf-json/`.
- **Виклик даних**: Використовувати `get_fields()`.
- **InnerBlocks**: Контент (текст, картинки, кнопки) має виводитись через `<InnerBlocks />`, а не через ACF, де це можливо. ACF використовується лише для налаштувань (Layout, Color, Options).

## 4. Стратегія редизайну та нових блоків
- **Сумісність**:
    - Якщо дизайн блоку змінюється мінімально (стилі/розміри) — модифікуйте `template.php` та додавайте класи.
    - Якщо дизайн кардинально інший (несумісний) — **створюйте новий блок** (наприклад, `HeroRoutes` замість `HeroBanner`). Це запобігає конфліктам логіки в адмінці.
- **Шаблонізація**: При роботі з різними варіаціями (наприклад, для одного блоку) використовувати клас `template-{N}`. Якщо логіка кардинально відрізняється — використовувати `if/else` у `template.php` або розділяти на два різні блоки.

## 5. Важливі нагадування
- **Vite/Build**: Зміни в `styles.scss` та `script.js` автоматично підхоплюються Vite.
- **Перевірка**: Завжди перевіряйте, щоб `get_block_wrapper_attributes()` включав динамічні класи (align, margin, template).
